# ReSale Agent 后续开发方向

## 当前阶段判断

ReSale Agent 当前已经达到 V3 本地闭环 MVP 阶段。项目已经具备完整演示链路：图片识别、用户确认、动态追问、本地相似样本检索、规则估价、出售文案、模拟议价、库存管理、发布后表现反馈、成交复盘、授权价格样本管理和审计导出。

接下来不建议继续盲目堆功能，也不建议立即进入真实平台自动发布、支付、物流和售后。更合理的方向是：先把 V3 原型打磨成稳定、清晰、可交付、可演示的产品化 MVP。

## 总体原则

- 继续保持本地可控闭环，不接真实平台登录和交易链路。
- 继续保持估价可解释，不让大模型直接决定价格。
- 继续保持模型可选，没有模型密钥时仍能跑通 Demo。
- 优先提升稳定性、可演示性、可维护性，再考虑新能力。
- 新增功能必须能被测试覆盖，不能只停留在界面展示。

## P0：产品化收口

P0 的目标是让项目更像一个可以交付给别人试用的 MVP，而不只是我们自己能跑的开发 Demo。

### 1. V3 验收文档

新增 `docs/v3-acceptance.md`，明确：

- V3 完成了哪些能力。
- 每个能力对应的演示路径。
- 每个能力对应的测试命令。
- 哪些能力明确不在 V3 范围内。
- 没有模型密钥时如何验收。

验收标准：

- 文档能让新协作者在 10 分钟内理解 V3 完成情况。
- 文档包含至少一条完整端到端演示路径。
- 文档明确写出不做真实平台登录、自动发布、支付、物流和售后。

### 2. 前端信息架构整理

当前 Streamlit 侧边栏已经承载草稿、成交复盘、库存管理和价格样本管理，信息密度偏高。建议整理为更清晰的页面或 tab：

- 出售流程：开始、确认、补充信息、出售方案、模拟议价。
- 复盘中心：成交反馈、成交汇总、库存总览。
- 数据源管理：价格样本导入、停用、审计导出。
- 项目说明：演示说明和数据边界。

验收标准：

- 用户第一次打开界面能清楚知道从哪里开始。
- 出售主流程不被数据源管理干扰。
- 价格样本管理仍然能完成导入、停用、恢复、删除和导出审计。

### 3. 启动和演示脚本收口

补齐一键演示说明和环境检查：

- 检查 Python 版本。
- 检查依赖安装。
- 初始化数据库。
- 启动后端。
- 启动前端。
- 跑固定演示案例。

可以新增脚本或补充 README，优先避免复杂部署工具。

验收标准：

- 新环境按 README 操作能稳定启动。
- `python scripts/run_demo_cases.py` 能稳定输出六个固定案例。
- 测试数据库不会污染本地正式 Demo 数据。

### 4. README 演示路线强化

README 目前覆盖启动、接口和测试，但演示路线还可以更具体。建议补一节“完整演示路线”，包含：

- 上传或使用模拟图片。
- 确认识别结果。
- 补充商品信息。
- 生成出售方案。
- 记录库存状态。
- 记录发布表现。
- 记录成交反馈。
- 导入价格样本并查看审计。

验收标准：

- 面试或演示时可以直接照着 README 走一遍。
- 每一步都能对应到界面或 API。

## P1：数据治理增强

P1 的目标是继续强化本地价格样本和复盘数据的可治理性，但不引入真实平台数据抓取。

### 1. 价格样本批量管理

当前价格样本支持单条停用、恢复、备注和删除。下一步可以支持批量操作：

- 批量按来源停用。
- 批量按类别停用。
- 批量恢复。
- 批量删除导入样本。
- 批量导出选中样本。

API 建议：

- `POST /api/v1/market-data/samples/batch`
- 请求中包含 `item_ids`、`action`、`user_notes`

验收标准：

- 只能批量操作用户导入样本，不能修改内置模拟样本。
- 批量操作需要写入审计记录。
- 批量停用后，样本不再参与相似商品检索。

### 2. 价格样本质量提示

对导入样本做轻量质量检查：

- 成交价高于原价过多。
- 成交价低于原价过多。
- 同品牌型号价格离群。
- 关键信息缺失。
- 同来源重复样本过多。

先做规则提示，不做复杂机器学习。

验收标准：

- 导入后返回质量提示。
- 样本列表展示异常标记。
- 异常样本可以一键停用。

### 3. 复盘数据导出

成交复盘和库存表现也可以导出：

- 成交反馈 CSV。
- 库存状态 CSV。
- 发布表现 CSV。
- Markdown 复盘报告。

验收标准：

- 导出内容包含时间、类别、商品、价格、状态和备注。
- 导出不依赖真实平台数据。

## P2：体验和工程化增强

P2 的目标是提升项目可维护性和可部署性，但避免过早复杂化。

### 1. 前端视觉和交互优化

当前 Streamlit 已满足功能演示，但可以优化：

- 页面布局更清晰。
- 表格字段更可读。
- 下载按钮和操作按钮位置统一。
- 错误提示更明确。
- 价格拆解和库存状态更易扫描。

验收标准：

- 常用流程点击更少。
- 页面不会把主流程和管理能力混在一起。
- 移动端不是重点，桌面演示体验优先。

### 2. 配置和环境检查

新增环境检查能力：

- 检查 `.env` 是否存在。
- 检查数据库路径是否可写。
- 检查上传目录是否可写。
- 检查可选模型配置是否完整。

验收标准：

- 配置错误时给出明确提示。
- 没有模型配置时提示“将使用本地降级”，而不是报错。

### 3. 数据迁移整理

当前 SQLite 通过初始化函数做兼容迁移。后续如果字段继续增多，可以考虑轻量迁移机制：

- 维护迁移版本表。
- 每次迁移记录版本号。
- 避免重复检查过多列。

验收标准：

- 老数据库升级不丢数据。
- 新数据库初始化不需要额外步骤。
- 测试覆盖至少一个旧库迁移场景。

### 4. API 文档和示例请求

FastAPI 自带 OpenAPI，但可以补充人工示例：

- 创建会话。
- 提交确认结果。
- 回答追问。
- 生成出售方案。
- 导入价格样本。
- 导出审计。

验收标准：

- 新开发者不看前端也能用 API 跑通主流程。
- 示例请求可以直接复制到 curl 或 HTTP client 使用。

## P3：谨慎探索真实平台适配

真实平台适配只能作为后续探索，不应直接进入自动化交易。

可以探索：

- 生成适配不同平台的字段草稿。
- 输出人工复制用的发布清单。
- 在用户明确授权下研究平台开放能力。
- 接入合规价格数据源，而不是爬取网页。

暂不做：

- 不做真实平台账号登录。
- 不做自动发布。
- 不做真实买家自动聊天。
- 不做支付、订单、物流、售后。
- 不做绕过平台规则的抓取和自动化。

验收标准：

- 所有平台相关能力必须以人工确认为前提。
- 所有数据来源必须可解释、可授权、可关闭。
- 不把项目变成违规自动化工具。

## 推荐下一步开发顺序

建议按下面顺序继续：

1. `docs/v3-acceptance.md`：先把 V3 验收说清楚。
2. README 完整演示路线：让项目更容易展示。
3. 前端信息架构整理：把主流程、复盘、数据源管理拆清楚。
4. 价格样本批量管理：补齐当前数据治理能力。
5. 复盘数据导出：让成交和库存数据也能备份。
6. 环境检查：提升新环境启动成功率。

## 面试叙述建议

如果面试官问“项目接下来怎么做”，可以这样回答：

> 当前项目已经完成 V3 本地闭环，接下来我不会急着接真实平台，而是先做产品化收口。第一步补 V3 验收文档和完整演示路线，保证项目可交付、可复现；第二步整理前端信息架构，让出售主流程、复盘中心和数据源管理更清晰；第三步继续增强价格样本治理，比如批量停用、质量提示和复盘导出。真实平台适配会放在更后面，而且必须基于授权和合规边界，不进入支付、物流和售后。

## 风险清单

- 功能继续增加后，Streamlit 单页可能变得拥挤。
- 样本导入能力如果缺少质量提示，可能影响估价稳定性。
- SQLite JSON state 适合 Demo 和 MVP，但生产阶段需要更规范的数据建模。
- 模型接入如果过度参与核心决策，会削弱可解释性。
- 真实平台适配容易触碰合规和平台规则，需要谨慎推进。

## 收口判断

如果只是作为面试项目或 Demo，当前 V3 已经可以收口。后续开发应该优先服务于“更好展示、更好验证、更好维护”，而不是继续追求大而全的自动化交易平台。
