# ReSale Agent 项目面试八股文

## 一句话介绍

ReSale Agent 是一个 AI 二手物品出售助手。用户上传闲置物品图片并补充必要信息后，系统会完成商品识别、动态追问、相似样本检索、规则估价、出售文案生成、议价回复、库存管理、发布后复盘和授权价格样本管理。

面试时可以这样说：

> 这是一个围绕二手物品出售场景设计的 Agent 项目。我没有直接做真实平台自动发布，而是先把本地可控闭环做完整：从图片识别、信息补全、估价、文案、议价，到库存、成交反馈、发布后表现和价格样本治理，形成一条可解释、可测试、可演示的出售辅助工作流。

## 项目定位

- 项目阶段：V3 本地闭环 MVP。
- 核心目标：帮助用户更高效、更稳妥地出售闲置物品。
- 产品边界：不登录真实二手平台，不爬取平台数据，不处理支付、物流、售后。
- 重点能力：Agent 流程编排、规则估价、本地样本检索、用户确认、数据闭环、可解释性和安全降级。

## 技术栈

- 后端：FastAPI、Pydantic、SQLite。
- 前端：Streamlit。
- Agent 编排：自定义状态节点和流程图。
- 模型能力：OpenAI 兼容多模态接口，可选接入图片识别和文本生成。
- 本地工具：字段检查、商品检索、价格计算、图片线索相似度、Markdown/CSV 导出。
- 测试：pytest、FastAPI TestClient、固定演示案例脚本。

## 整体架构

项目按职责分层：

- `frontend/`：Streamlit 前端，负责上传图片、确认信息、补充字段、查看方案和管理库存/样本。
- `backend/app/api/`：FastAPI 路由层，负责请求校验和服务编排。
- `backend/app/agent/`：Agent 状态机节点，负责串联识别、追问、估价、文案等步骤。
- `backend/app/services/`：业务服务层，例如估价、文案、议价、库存、导入导出。
- `backend/app/tools/`：确定性工具，例如字段检查、价格计算、相似商品搜索。
- `backend/app/repositories/`：SQLite 数据访问层。
- `backend/tests/`：单元测试和完整 API 流程测试。

这种分层的好处是：Agent 负责流程，工具负责确定性计算，服务负责业务编排，模型只做辅助生成，不让模型直接控制核心价格和交易决策。

## Agent 主流程

核心链路是：

1. 创建出售会话。
2. 上传图片并生成识别初稿。
3. 用户确认或修正识别结果。
4. 系统检查缺失字段。
5. 动态追问必要信息。
6. 检索本地相似商品样本。
7. 规则估价并校验价格边界。
8. 生成出售标题、描述、关键词、拍照建议和多平台文案。
9. 模拟买家议价回复。
10. 记录库存、发布表现、成交反馈和复盘数据。

面试回答重点：

> 我把 Agent 设计成状态驱动，而不是一次性让大模型输出所有东西。每个节点只做一类任务，节点之间通过 state 传递结构化数据。这样方便测试、恢复草稿、定位错误，也方便在模型失败时降级。

## 为什么不用大模型直接估价

估价是项目里比较敏感的决策点。我的做法是：

- 大模型可以辅助识别图片和生成文案。
- 价格由确定性规则计算。
- 估价依据包括原价、使用时长、成色、功能状态、维修记录、配件情况和本地相似成交样本。
- 价格结果会做边界校验，例如挂牌价不能低于成交区间上限，最低接受价不能高于挂牌价。

面试回答可以这样说：

> 价格不直接交给大模型，是因为大模型容易幻觉，而且价格需要可解释、可复盘。我用规则估价保证稳定性，用本地样本中位数增强市场参考，再把估价拆解展示给用户。模型只负责非关键决策的文本生成。

## 动态追问怎么做

系统不会一开始就问所有字段，而是按商品类别检查必填信息：

- 数码：原价、购买时间、功能状态、维修记录、配件等。
- 图书：套装情况、笔记情况、破损情况等。
- 家具：尺寸、拆装、搬运、清洁状态等。
- 鞋包：尺码、磨损、清洁、验货信息等。

字段检查工具会输出缺失字段，Agent 每次只追问当前最关键的一项。用户回答后更新 state，再继续检查。

面试回答重点：

> 动态追问本质上是一个表单补全问题。我没有完全依赖大模型判断缺失信息，而是按品类维护结构化字段规则。这样能保证信息完整性，也方便测试每个品类的必填字段。

## 相似商品检索怎么做

相似商品来自本地 SQLite 样本库，默认包含六类模拟数据，也支持用户导入授权 CSV 成交样本。

检索评分包括：

- 商品类型匹配。
- 品牌匹配。
- 型号匹配。
- 本地图片线索匹配。
- 用户导入样本标记。

图片线索不是连接真实平台图片库，而是用本地上传图片的文件名、签名和识别线索辅助排序。

面试回答重点：

> 我把真实平台数据接入放在后续合规阶段。当前版本只使用本地模拟数据和用户授权导入样本，既能演示相似样本融合，也避免爬虫和平台风控问题。

## 估价逻辑怎么解释

估价结果包括：

- 建议挂牌价。
- 合理成交区间。
- 最低接受价。
- 估价置信度。
- 估价原因列表。
- 调价明细。

调价明细会解释：

- 原价影响。
- 折旧影响。
- 成色影响。
- 功能/维修/配件影响。
- 相似成交样本融合影响。
- 最终挂牌建议。

面试回答可以这样说：

> 我把估价从一个黑盒数字拆成多个阶段，让用户能看到每一步为什么涨价或降价。这能提升用户信任，也方便后续成交反馈反推规则是否合理。

## 议价回复怎么保护底价

议价模块会从买家消息里提取报价，并和用户最低接受价比较：

- 如果报价低于最低接受价，只建议拒绝或继续沟通。
- 如果报价在安全范围内，再生成更自然的沟通话术。
- 文本模型可以辅助润色，但不能建议低于最低接受价成交。

面试回答重点：

> 我把最低接受价作为硬约束放在业务逻辑里，而不是写在提示词里。提示词可以增强表达，但不能代替规则约束。

## V2 做了哪些增强

V2 主要围绕“更可解释、更可复用的出售辅助”：

- 多品类扩展：服装、家具、鞋包。
- 草稿管理：保存、恢复、删除出售会话。
- 多平台文案：闲鱼、转转、小红书风格文案。
- 成交反馈：记录最终成交价、成交渠道和备注。
- 成交复盘：按类别和渠道统计命中比例与偏差。
- 本地图片相似度原型。
- 调价明细展示。
- 发布前检查清单。

面试回答重点：

> V2 的重点不是堆更多模型能力，而是把出售辅助做成可解释、可复盘、可再次打开的工具。

## V3 做了哪些闭环

V3 主要围绕“本地数据闭环和治理”：

- 个人库存：草稿、待发布、已发布、已成交、已归档。
- 库存总览：按类别和状态查看闲置物品。
- 发布后表现：发布天数、浏览量、收藏量、咨询量、当前挂牌价。
- 本地调价建议：继续观察、保持价格、优化展示、小幅降价，并守住最低接受价。
- 授权价格样本导入：用户上传 CSV，增强本地样本库。
- 价格样本管理：查看、筛选、停用、恢复、备注、删除。
- 价格样本审计：记录导入、停用、恢复、删除等操作。
- 审计导出：CSV 和 Markdown。

面试回答重点：

> V3 的核心是让项目从一次性生成方案，变成一个可以持续复盘和校正的数据闭环。用户每次成交、每次导入样本、每次调整价格，都会沉淀成本地数据。

## 价格样本为什么要做治理

如果允许用户导入样本，就必须考虑数据质量：

- 样本可能重复。
- 样本可能价格异常。
- 样本来源可能需要追溯。
- 样本可能需要暂时停用，而不是直接删除。

所以项目里做了：

- 重复样本跳过。
- 导入行级错误记录。
- 样本来源字段。
- 启用/停用状态。
- 用户备注。
- 删除限制：内置样本不能删除。
- 审计记录：删除后仍保留样本快照。
- 审计导出：方便备份和复盘。

面试回答重点：

> 只做导入是不够的。真实业务里数据质量会直接影响估价，所以我补了样本治理能力：停用、备注、审计、导出。这样既能使用用户数据，又能保留可追溯性。

## 安全降级怎么做

系统设计上允许没有模型密钥：

- 图片识别失败时，返回可编辑的结构化初稿。
- 文案生成失败时，使用本地模板。
- 议价回复模型失败时，使用规则回复。
- 核心估价不依赖模型。

面试回答重点：

> 我把模型能力设计成增强项，而不是系统唯一入口。即使没有 API Key，整条 Demo 链路也能跑通，这对演示、测试和本地开发都很重要。

## 数据库设计

核心表包括：

- `sale_sessions`：出售会话状态，保存 Agent state。
- `reference_items`：本地价格样本，包含内置模拟样本和用户导入样本。
- `market_sample_events`：价格样本操作审计。
- `negotiation_messages`：议价消息记录。

为什么会话状态用 JSON 保存：

- Agent state 字段会随版本扩展。
- Demo 阶段更需要快速迭代。
- 恢复草稿和导出报告时读取完整状态更方便。

面试中需要补一句：

> 如果进入生产阶段，可以把核心实体拆成更规范的关系表，同时保留 state 快照用于审计和回放。

## API 设计

主要接口分几类：

- 会话：创建、查询、删除、恢复。
- 图片：上传并分析。
- 信息补全：确认识别结果、获取下一题、提交回答。
- 出售方案：生成 listing、导出报告。
- 议价：生成模拟回复。
- 库存：更新库存状态、记录发布表现。
- 成交：记录成交反馈、查看复盘汇总。
- 价格样本：导入、列表、更新、删除、审计、导出审计。

面试回答重点：

> API 不是按模型能力划分，而是按业务资源划分。比如 sessions 管出售会话，market-data 管价格样本，listing 管出售方案。

## 项目亮点

可以重点讲这几个：

- 没有把价格交给大模型，而是规则和样本融合，保证可解释。
- Agent state 可恢复，支持草稿和完整报告导出。
- 多品类字段追问，体现业务建模能力。
- 发布后表现反馈和成交反馈，形成数据闭环。
- 授权价格样本导入和治理，体现数据质量意识。
- 模型能力安全降级，没有 API Key 也能演示。
- 严格划定边界，不接真实平台、不爬数据、不碰支付物流。

## 项目难点

### 难点一：怎么让 Agent 不失控

解决方案：

- 状态机拆节点。
- 每个节点只做单一职责。
- 核心决策用规则。
- 模型输出结构化校验。
- 用户确认优先级高于模型识别。

### 难点二：怎么让估价可信

解决方案：

- 原价、折旧、成色、功能、维修、配件分阶段计算。
- 相似样本只做融合参考。
- 输出价格拆解。
- 成交反馈用于复盘。
- 用户导入样本可以停用和审计。

### 难点三：怎么处理模型不可用

解决方案：

- 模型只是增强项。
- 图片识别、文案、议价都有本地降级。
- 测试优先覆盖降级链路。

### 难点四：怎么避免项目变成真实平台自动化

解决方案：

- 明确不做真实登录、自动发布、真实聊天、支付、物流、售后。
- 只生成发布建议和文案。
- 价格数据只接受本地模拟数据和用户授权导入。

## 高频面试问答

### Q1：这个项目和普通 CRUD 有什么区别？

A：它不是单纯 CRUD，而是一个带 Agent 流程的业务闭环。系统会根据图片识别和用户输入动态判断缺失信息，再检索本地样本、规则估价、生成文案和议价建议，后续还能记录库存、发布表现和成交反馈。

### Q2：为什么不用 LangChain 或 LangGraph？

A：当前项目流程比较明确，核心需求是可控、可测试和易解释。我用轻量自定义状态节点就能满足需求，避免过早引入复杂框架。如果后续需要更复杂的工具调用、并发节点或可视化编排，可以再迁移到 LangGraph。

### Q3：估价准确吗？

A：当前定位是辅助估价，不承诺等于真实成交价。系统通过规则和本地样本给出相对合理的区间，并展示估价依据。后续如果接入合规价格数据源，准确性可以继续提升。

### Q4：如果模型识别错了怎么办？

A：模型识别只是初稿，用户确认结果会覆盖模型输出。系统后续估价和文案都使用用户确认后的结构化字段。

### Q5：价格样本为什么要支持停用？

A：用户导入的数据可能有异常值。直接删除会丢失追溯信息，所以支持先停用，让样本不参与估价，但保留记录和备注。如果确认无效，再删除，同时审计表仍保留删除前快照。

### Q6：为什么内置样本不能删除？

A：内置样本保证 Demo 和测试稳定。如果允许删除，用户可能把基础样本删掉，导致演示链路不稳定。用户导入样本才允许修改和删除。

### Q7：项目如何保证测试稳定？

A：核心链路可以不依赖真实模型。测试数据库放临时目录，图片识别和文案可以走本地降级，固定演示案例确保主流程输出稳定。

### Q8：如果要上线生产，需要补什么？

A：需要补用户账号、权限、真实文件存储、任务队列、结构化数据库迁移、可观测性、部署配置、合规数据源和更完整的安全策略。但自动发布、支付、物流、售后仍然要谨慎推进。

### Q9：这个项目最大的设计取舍是什么？

A：最大取舍是没有追求“全自动卖货”，而是先做“可控辅助”。这让项目更稳，能在不触碰平台风险的情况下完成识别、估价、发布辅助、议价建议和复盘闭环。

### Q10：你在这个项目里最想强调的工程能力是什么？

A：我想强调三点：第一，能把模糊业务拆成可执行的 Agent 状态流；第二，知道哪些地方该用模型，哪些地方必须用规则；第三，能把一次性功能继续做成数据闭环和可治理系统。

## 30 秒口述版本

这是一个 AI 二手物品出售助手。我用 FastAPI、Streamlit 和 SQLite 做了一个本地可演示的 Agent 工作流：用户上传图片后，系统识别商品、动态追问缺失信息、检索本地相似样本、用规则估价、生成出售文案和议价回复。后续我又补了库存管理、发布表现反馈、成交复盘，以及授权价格样本导入、停用、审计和导出。这个项目的重点不是让大模型直接决定价格，而是把模型能力放在识别和文本生成上，核心估价和底价保护都用确定性规则，保证可解释、可测试和可控。

## 2 分钟口述版本

ReSale Agent 是我做的一个二手物品出售辅助 Agent。它的主流程是用户上传闲置物品图片，系统给出识别初稿，用户确认后，Agent 根据商品类别动态追问缺失字段，比如数码要问功能、维修、配件，家具要问尺寸、搬运，鞋包要问尺码和验货信息。信息完整后，系统会检索本地相似样本，用规则估价生成挂牌价、成交区间和最低接受价，并输出价格拆解、出售文案、拍照建议、多平台风格文案和议价回复。

这个项目里我刻意没有让大模型直接决定价格，因为价格需要稳定和可解释。我把模型作为增强项，用在图片识别、文案和议价润色上；核心估价、底价保护和字段检查都用确定性逻辑实现。没有模型密钥时，项目也能通过本地降级链路跑完整 Demo。

后续我把项目从一次性生成方案扩展成 V3 本地闭环：支持草稿、库存状态、发布后浏览/收藏/咨询反馈、成交复盘，还支持用户导入授权成交样本来增强本地样本库。为了避免导入数据污染估价，我又做了样本停用、恢复、备注、删除、审计记录和审计导出。整体上，它是一个可演示、可测试、边界清楚的 AI 出售助手原型。

## 简历写法

- 设计并实现 AI 二手物品出售助手 ReSale Agent，覆盖图片识别、动态追问、相似样本检索、规则估价、文案生成、模拟议价、库存管理和成交复盘。
- 基于 FastAPI、Streamlit、SQLite 构建本地可演示闭环，采用自定义 Agent 状态节点拆分识别、信息补全、估价、文案和议价流程。
- 将核心估价设计为确定性规则和本地样本融合，输出价格拆解和置信度，避免模型直接参与价格决策。
- 实现授权价格样本导入、停用、恢复、备注、删除、审计和导出能力，提升估价数据源的可治理性。
- 建立完整测试覆盖，包含字段检查、估价规则、相似检索、API 主链路、样本治理和固定演示案例。
